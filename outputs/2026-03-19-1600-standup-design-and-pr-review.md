> **Question:** I want to keep a daily standup schedule: script creates initial context, LLM creates a summary (which agent?), agents contribute. Add a mechanism to only wake the LLM/agents if the initial context has changed. Each standup note should report credit consumed. Also: what is the preferred way to review a PR — via editing the PR or via the chat window? If I respond via the chat window, what happens to the initial PR — will the agent change it or create a new one?

## Part 1: Standup Design

### The full flow

```
07:30 Mon–Fri
    │
    ▼
[cron script]  ← no LLM
  Generate context:
    - git log across all repos (commits since yesterday)
    - open PRs across all repos (GitHub API)
    - BACKLOG.md diff (tasks moved, added, closed)
    - new files in proposals/, discussions/
  Hash the content
    │
    ├─ hash == yesterday's hash?
    │       └─ YES → write standups/YYYY-MM-DD.md with
    │                 "No changes since yesterday. Standup skipped."
    │                 Stop. No LLM woken.
    │
    └─ hash changed?
            └─ Write raw context to standups/YYYY-MM-DD.md
               Wake Quality Manager (cron job) →
                   QM synthesises summary section →
                   appends to standup file
               CEO arrives, opens each agent tab:
                   "Contribute to today's standup"
```

### Which agent writes the summary?

**Quality Manager (Veri).**

The summary is a cross-project synthesis: what changed in each repo, which PRs are open or stale, what moved on the backlog. This is exactly QM's existing remit — reviewing state across all repos with a critical eye. The standup summary is just a lighter version of that.

The alternative would be programme manager (Marco), but Marco's domain is field coordination and comms — not codebase state. QM has the right context and the right role.

Practical benefit: QM writing the summary before the other agents contribute means the agents that follow have a synthesised picture, not just raw data. Later agents can respond to QM's framing.

### Change detection — what to hash

The script should hash the union of:
- `git log --oneline --since="yesterday 07:00"` across all repos
- Open PR titles + states from GitHub API (engine-dev, console-dev, site-dev, quality-manager, programme-manager repos)
- `BACKLOG.md` file content (or just its last git commit hash)
- Filenames in `proposals/` and `discussions/` (new items signal something worth discussing)

Store the hash at `standups/.last-context-hash`. If today's hash matches, skip everything.

**Important edge case:** if it's Monday and the hash matches Friday's, there may still be value in a standup (start-of-week orientation). Consider: on Mondays, always wake regardless of hash, but mark the standup as "no codebase changes since Friday."

### Credit reporting

This is where OpenClaw and the Claude API create a practical constraint: session-level token usage is visible in Claude Code's status line and in the Anthropic console, but there is no built-in mechanism to automatically write per-session token counts into a file.

**What is feasible now:** each agent appends a `## Session cost` block to their standup contribution, self-reported:

```markdown
## Session cost
- Input tokens (est.): ~4,200 (HEARTBEAT.md + AGENTS.md + standup context + task state)
- Output tokens (est.): ~380
- Tool calls: 4 (git log, 2× file reads, standup write)
- Model: claude-opus-4-6
```

These are estimates — the agent knows roughly how much context it loaded and how long its response was. Not exact, but useful for spotting expensive outliers.

**What would give exact numbers:** a wrapper script that calls the Claude API directly (not through OpenClaw) and captures `usage.input_tokens` + `usage.output_tokens` from the response. This is viable for the summary step (a single API call), but harder for the full agent sessions which involve many tool call turns.

**Recommended approach for now:**
1. Summary step (QM): exact cost is capturable if invoked via a direct API call from the cron script. Write the token counts into the standup header.
2. Agent contributions: self-reported estimates. Better than nothing; good enough for spotting if a standup is unexpectedly expensive.
3. Standup file footer: a manual tally section the CEO fills in: "Today's standup: QM summary + N agents = approx $X based on Anthropic console."

The more precise solution — instrumenting all sessions to emit token counts — is a backlog item, not something to implement before the standup system exists.

## Part 2: PR Review — Chat vs. GitHub

### What actually happens when an agent opens a PR

The agent creates a feature branch (e.g. `feature/usage-analytics`), commits its changes to that branch, and runs `gh pr create` to open a pull request on GitHub. The PR points to that branch. The agent's workspace has the branch checked out.

**The PR is not a snapshot. It is a live view of the branch.** If the agent pushes more commits to the same branch, the PR updates automatically — same PR number, same URL, new diff.

### Option A: Review via GitHub (comments and suggestions)

1. You open the PR on GitHub
2. Leave inline comments ("this function is doing two things, split it") or use GitHub's suggestion feature to propose exact replacement text
3. Request changes (GitHub's formal review mechanism)
4. Open the agent's MC chat: *"I've reviewed your PR — please address the comments"*
5. Agent reads the PR via tools (`gh pr view`, `gh pr diff`, reads comment thread)
6. Makes the requested changes
7. Commits and pushes to the **same branch**
8. PR updates automatically — same URL, same PR, the diff now shows the revised version
9. You review again on GitHub, approve, merge

**No new PR is created.** The agent is working on the branch the PR tracks. Pushing to that branch is how you update a PR — this is standard git/GitHub behaviour and the agent knows it.

### Option B: Review via chat only (no GitHub comments)

1. You look at the PR on GitHub (or read the diff in chat)
2. You describe what needs changing in the MC chat window: *"The error handling in line 47 is too broad — catch only the specific error types"*
3. Agent implements the changes, commits, pushes to the same branch
4. PR updates

This works fine. The difference from Option A is that the review conversation lives in MC chat, not in the GitHub PR thread. That is a meaningful audit trail trade-off: six months later, the GitHub PR shows only the commits, not the reasoning behind each change.

### Preferred approach: GitHub-first, chat for context

**Leave review comments on GitHub. Then tell the agent in chat.**

```
You (GitHub):  [inline comment on line 47] "Too broad — catch specific errors only"
You (MC chat): "I've left comments on PR #14. Please address them."
Agent:         [reads PR, reads comments, makes changes, pushes to branch]
GitHub:        [PR automatically updates with new commits]
You (GitHub):  [review the new diff, approve, merge]
```

Why this is better:
- **Audit trail.** The PR thread records *what* you asked for and *what* the agent changed. Visible to anyone who reads the PR later, including future agents reviewing the code.
- **Inline precision.** GitHub's inline comment UI pins your feedback to exactly the right line. Describing line numbers in chat is imprecise.
- **GitHub suggestions.** You can propose exact replacement text; the agent can accept it with one tool call (`gh pr review --comment` or by editing the file to match). Avoids misinterpretation.

Chat-only review is fine for minor feedback ("change the variable name") but loses the audit trail for anything more substantive.

### What the agent will NOT do

- It will not create a new PR for the same changes. Doing so would mean creating a new branch, leaving the old PR open, and creating a confusing parallel history. Agents know to push to the existing branch.
- It will not merge the PR. Branch protection + CEO-only merge enforces this mechanically.
- It will not close the PR during revision. It keeps it open and updates it.

### One edge case: the review reveals the approach was wrong

If your review feedback is not "fix these details" but "this whole approach is wrong — start over," the agent should:
1. Close the existing PR with an explanation
2. Create a new branch for the new approach
3. Open a new PR

This should be an explicit instruction from you — the agent won't infer "start over" from detailed critique alone. If the critique is extensive, it will try to patch the existing PR. Say explicitly: *"Close this PR and start fresh with a different approach."*

## Summary

**Standup:** daily cron script, hash-based skip if nothing changed, Quality Manager writes the synthesis, agents contribute sequentially. Cost reported in a table at the bottom of each standup file — QM summary can be exact; agent contributions are self-estimated.

**PR review:** leave comments on GitHub (inline, with suggestions where applicable), then tell the agent in MC chat. The agent pushes to the same branch, the PR updates automatically. No new PR is created. Chat-only is fine for minor changes but loses the audit trail.
